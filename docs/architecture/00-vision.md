# 00 愿景与设计原则（Vision）

Status: Accepted Direction（已确认方向，非冻结契约）

> 本文是编号架构系列（00–07）的开篇。系列是平台架构视图（platform view）：除本篇与
> [07 演进路线](07-roadmap.md) 的方向性内容外，不定义任何新契约；与冻结契约文档冲突时，
> 以 [核心架构](AICODING_CORE_ARCHITECTURE.md) 等契约文档为准。方向性内容的落地
> 仍须走 [06 扩展规范](06-plugin-sdk.md) 的路径与 ADR 流程。

## 本篇回答的问题

- 这个平台要做成什么样？（愿景）
- 凭什么这样分层、这样冻结？（设计原则）
- 未来的不确定性怎么管理？（四象限演进观）

## 1. 一句话愿景

> 让 AI Agent 在本机安全、可验证地写代码：仓库不仅提供**规则**（门禁、政策、工作流），
> 还提供**随代码演进持续更新的上下文**（Agent 所需的项目知识），两者由同一个
> Go 控制面管理。

## 2. 三者结合：能力版图

平台愿景对照两个已验证的生态项目定位（均为 MIT 许可，以纯文本坐标引用，不并入源码）：

| 能力层 | 生态参照 | AiCoding 现状 | 结论 |
|---|---|---|---|
| 控制面：CLI、生命周期、Hook、Release、治理 | —（AiCoding 本体） | 已冻结的 Go CLI 地基，见 [01](01-system-architecture.md) | **保留，不动** |
| 工作流纪律：先计划后执行、TDD、评审、验证 | `Bollwerkio/werkstatt`（Superpowers 分支） | plugin 内 SDD/BDD/TDD/计划模式（Plan Mode）技能 + 门禁 hook 已覆盖；Superpowers 系定位为可选加速（`AGENTS.md` 既有立场） | **已有对应物**；个别缺失技能按外部 Skill 路径吸收，见 [07](07-roadmap.md) §4 |
| 仓库上下文：从代码自动生成、随提交自动更新的 Agent 上下文 | `aspenkit/aspens` | 只有雏形（`config/repository-navigation.json` 生成 REPOSITORY_MAP 区块）；无自动生成、无增量同步 | **当前缺失的一层**，立项为新领域能力 repo-context，见 [07](07-roadmap.md) §3 |

一句话：**规则层已经收敛，上下文层是下一步。** 仓库从"只提供规则"升级为
"规则 + 随代码演进的上下文"。

## 3. 四象限演进观：地基就是已知

平台用"已知/未知"四象限管理确定性。功能沿"未知的已知 → 已知的未知 → 地基"
单向沉淀，地基只进不出：

| 象限 | 本仓库对应物 | 治理机制 | 详见 |
|---|---|---|---|
| 已知的已知（**地基**） | 内核六模块、八个 action 动词、内核命令面、JSON 报告契约、三档测试 | 冻结 + 可执行门禁；一切新功能站在它上面 | [01](01-system-architecture.md) |
| 已知的未知 | 已识别、有触发条件的未来功能（repo-context、流式执行、并发写守卫……） | 预留出口 + 触发条件；不预建 | [07](07-roadmap.md) §2–§3 |
| 未知的已知 | 生态已验证、本仓库尚未吸收的能力（werkstatt 技能、aspens 概念、外部 Skill） | 获取/激活分离边界 + 吸收路径①② | [07](07-roadmap.md) §4 |
| 未知的未知 | 无法预测的 AI 变化（新工具、新协议、模型更迭、分发形态） | 不预测，靠吸收层兜底：变化落在登记层/adapter 层/调用方层，内核零改动；解冻走 ADR + 三条件 | [07](07-roadmap.md) §5 |

### 3.1 每个象限沉淀什么（可沉淀的知识资产）

四象限不只是分类，每个象限还对应一类**持续沉淀的知识资产库**：

| 象限 | 沉淀资产库 | 本仓库对应物 |
|---|---|---|
| ①已知的已知 | 上下文库（背景、常识、项目地图） | REPOSITORY_MAP 生成区块 → 未来 repo-context（[07](07-roadmap.md) §3） |
| ②已知的未知 | 决策库（问题清单、方案取舍、判断框架） | `.aicoding/memory/DECISIONS.md` + `docs/decisions/` 计划工件 |
| ③未知的已知 | 规则/偏好库（隐性标准显性化，永久免除重复） | governance 门禁 + `config/` 政策 + style 配置 |
| ④未知的未知 | 边界库（能力天花板、盲区地图、新发现） | 拒绝清单 + 冻结边界文档 + 预留出口表（[07](07-roadmap.md) §2） |

衡量复利的唯一硬指标：**已知的已知越来越厚，调用成本越来越低**——这正是
"地基只进不出 + 知识进入点恒定"（原则 5）的度量表述。

## 4. 设计原则（全系列共用）

1. **少量稳定原语 + 自由组合**：内核是六个正交模块，不是一个什么都懂的 Core
   （对照 Git 的 blob/tree/commit/ref）。
2. **只增不改**：动词、契约字段、命令形态一旦冻结，新能力以新条目进入，不改旧语义
   （`git add` 二十年未变）。
3. **不预建**："未来可能需要"不是理由；空接口与兼容层是债务。真实需求出现、
   三条件齐备才解冻。
4. **证据可对账**：每次执行都能回答"对什么事实、用什么契约、执行了什么意图"
   （digest 三元组 + JSON 报告，见 [04](04-workflow-architecture.md)）。
5. **知识进入点恒定**：功能数量可以无限增长，Agent 获取知识的入口数量保持恒定
   （见 [02](02-context-architecture.md)）。
6. **具体优先**：面向人的描述用大白话——每条命令一两句"它动了什么、输出什么"，
   像 git 基础命令一样直白。
