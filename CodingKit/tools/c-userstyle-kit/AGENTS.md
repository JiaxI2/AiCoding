# C UserStyle Kit Agent Instructions

## 适用范围与权威来源

本文件适用于本仓库以及由本仓库生成的 C99 黄金示例。实现或评审 C/H 文件前必须同时读取：

- `references/huawei-c-language-programming-standard-dkba-2826-2011-5.md`：61 页 PDF 的可检索参考副本；
- `docs/RULE_CATALOG.md`：139 条原则、规则和建议的完整证据目录；
- `examples/c-kit.json`：机器可执行配置；
- `examples/c-snippets.json`：最终用户可直接编辑或复制到 VS Code 的代码片段目录；
- 当前目录下更具体的 `AGENTS.md`（如果存在）。

PDF/Markdown 是规范含义的来源，规则目录用于检索和追踪，JSON、lint、编译与测试门禁用于执行。
无法可靠机器判断的语义规则不得伪装成正则检查，必须标为人工评审证据。

## 修改工作流

1. 先从规则目录定位受影响条款和现有证据。
2. 只修改与需求直接相关的模块；保持根 `demo.c/.h` 简单，把完整规则覆盖职责放在公开的
   `generated-demo/advanced/` 中。
3. 同步维护代码周边注释；删除已经失效的注释，不保留历史代码注释块。
4. 运行 `go test ./...`、规则目录检查、lint、GCC、Clang、头文件和行为测试门禁。
5. 失败必须修代码或配置，禁止降低告警级别、跳过负例或删除失败测试。

## 头文件与文件组织

- 每个公开实现文件必须有同名头文件：根目录 `demo.c/.h`，以及 `advanced/` 中的
  `state_machine.c/.h`、`protocol.c/.h`、`fixed_pool.c/.h`。
- 头文件只放稳定公开接口、必要类型和编译期常量，不放函数实现或变量定义。
- 头文件必须职责单一、自包含、使用传统 `#ifndef/#define/#endif` 保护宏，并独立通过 C99 与 C++17 编译。
- `#include` 只保留实际使用项；源文件首先包含自己的同名头文件。
- 禁止循环包含，禁止在 `.c` 中用手写 `extern` 绕过公开头文件。
- `extern "C"` 只包围 C 接口声明；禁止在该块内包含其他头文件。
- 文件名使用小写和常规 `.c/.h` 扩展名。

## 函数设计与声明

- 一个函数只完成一项清晰功能；重复逻辑提取为有实际复用价值的私有函数。
- 新增函数有效代码不超过 50 行，代码块嵌套不超过 4 层，参数不超过 5 个，合理扇出小于 7。
- 不变输入使用 `const`；无参数函数写成 `(void)`；禁止普通业务接口使用可变参数。
- 源文件内不对外公开的函数必须是 `static`，并使用模块命名规则。
- 静态函数前置区只登记函数原型，禁止在前置声明上堆叠 Doxygen：

```c
static bool DEMO_IsReady(const demo_context_t *context);
```

- 静态函数和公开函数的完整实现说明必须紧邻函数定义上方；公开声明的使用契约保留在头文件。
- 可重入函数避免共享可变状态；必须共享时明确单写者、临界区或其他互斥协议。
- 公开入口检查运行时参数并返回明确错误码；调用方必须处理全部错误返回。
- `assert` 只记录已经由程序逻辑保证的内部不变量，不能替代运行时错误处理。
- 删除本次变更造成的未使用函数和变量；不要保留废弃代码。

## 命名

- 名称必须表达功能，优先完整英文单词和业界通用缩写，禁止汉语拼音和无意义编号。
- 保持既有项目命名体系；平台适配代码遵循平台已有命名风格。
- 文件、局部变量、参数和成员使用 `lower_snake_case`；typedef 类型以 `_t` 结束。
- 全局变量使用 `g_`，文件静态变量使用 `s_`；黄金示例原则上不引入可变全局变量。
- 宏和枚举常量使用 `UPPER_SNAKE_CASE`，不得使用标准保留标识符形式。
- 普通变量不得使用单字符名，只有局部短循环可使用 `i`、`j`、`k`。
- 函数名使用动作或“动作+对象”；黄金示例公开与私有函数均使用 `DEMO_` 加 PascalCase。

## 变量、类型、宏与常量

- 每个变量只有一个用途，每个结构体只承载一个职责；尽量不使用全局变量。
- 变量在靠近首次使用处初始化，禁止读取未初始化值；跨模块初始化不得形成隐式顺序依赖。
- 接口和持久数据使用 `<stdint.h>` 固定宽度类型；长度、容量和字节数使用 `size_t`。
- 通信数据明确网络字节序，不直接把外部字节数组强制转换为结构体。
- 减少隐式转换；窄化、符号变化和移位前必须证明范围并显式表达。
- 不直接写魔法数字，使用带语义的枚举常量或 `const`。
- 优先函数或类型化常量；必须使用函数式宏时，参数和完整表达式都加括号。
- 多语句宏使用 `do { ... } while (0)`，宏参数不得产生自增等副作用，宏内避免改变外部流程。

## 质量、安全与效率

- 优先级依次为正确、简洁、可维护、可靠、可测，再考虑性能。
- 所有数组和缓冲区在访问前检查边界；循环条件采用半开区间，特别审查差一错误。
- 禁止动态内存、变长数组和无界循环；资源使用固定池并明确申请、读写、释放和复用生命周期。
- 固定资源句柄必须能拒绝重复释放和释放后访问；资源复用前清理旧数据。
- 整数加减乘除、移位、符号转换和窄化必须进行上溢、下溢、符号和截断分析。
- 循环不变量移到循环外；数组按连续存储顺序访问；只在有测量证据时采用内联或宏优化。
- 所有 `if/else if` 链应有明确结束处理；每个 `switch` 必须有 `default`。
- 禁止 `goto`；所有控制语句使用花括号。

## 注释（规范第 8 章重点）

- 注释解释意图、理由、约束、边界和并发语义，不逐字复述代码。
- 复杂函数先在定义处的 `@details` 中用 `1.`、`2.`、`3.` 等编号概述主要控制流；编号应对应
  参数校验、遍历或分派、结果发布等真实职责，不强制使用“阶段”之类固定措辞。
- 相对独立的业务逻辑块使用“空行 + 同缩进的 `/* 意图、约束或副作用 */` + 代码”；不要给
  每条赋值或自增补写重复代码含义的注释。
- 非显然的 `if/else` 分支说明领域条件的意义以及分支会保留或改变什么。简单参数 guard、立即
  返回和名称已经自解释的判断可以只用一条共享意图注释，不为满足形式制造空 `else`。
- 每个包含实际处理的 `case/default` 在首条语句前说明模式、领域意义或副作用。连续空 `case`
  标签不强制逐个注释；有意贯穿必须在相邻标签之间明确说明原因。
- 文件头必须包含：`@file`、`@brief`、版权、版本、日期、作者、文件内容、主要功能和文件关系。
  只有已获得真实工号时才写 `@employee_id`，禁止生成“不适用”或“未提供”占位行。
- 代码内修改历史默认禁用，变更记录由 Git 和 `CHANGELOG` 管理。只有维护发布、缺陷修复等场景
  经用户显式确认后，才允许在源文件中启用修改历史。
- 公开函数声明必须说明功能、用法、所有输入输出、返回值、性能/执行上界和可重入、并发或中断约束。
- 函数定义使用 `@details` 说明实现步骤、采用该实现的理由和设计约束；不得把完整说明复制到静态前置声明。
- 全局变量若确有必要，必须说明功能、合法范围、所有权、初始化顺序、读写者和互斥要求。
- 注释与目标代码相邻并位于其上方或右侧；上方注释与前一代码块留空行，并保持相同缩进。
- 若确需贯穿到下一 `case`，必须在两个标签之间写明确的贯穿原因；禁止无注释隐式贯穿。
- 首次使用专业缩写时给出中文含义和英文全称，例如直接内存访问
  （Direct Memory Access，DMA）和中断服务程序（Interrupt Service Routine，ISR）。
- 文件、公开接口、复杂函数、类型和全局数据使用 Doxygen；函数内部逻辑和简单对象式宏优先使用
  单行普通块注释 `/* ... */`。禁止 `//`，禁止在表达式中间插入注释。
- 注释符与正文之间保留一个空格；修改代码时同步更新或删除失效注释。
- 完整层级、正反例和例外见 `docs/COMMENTING_METHOD.md`；注释准确性属于人工评审，不能仅凭
  “存在注释”判定合规。

## 排版与表达式

- 缩进 4 个空格，不使用制表符；Allman 花括号；指针星号靠变量名。
- 变量声明后、相对独立的程序块之间留空行；一行只写一条语句。
- `if`、`for`、`while`、`switch`、`case` 和 `default` 独占一行；行宽上限 100。
- 二元操作符两侧留空格，成员访问等紧密一元操作保持常规格式。
- 表达式不得依赖未规定求值顺序；有副作用的函数调用和自增不得在同一表达式中重复组合。
- 不把函数调用嵌套为另一个函数参数，不在条件中赋值，用括号明确非显然优先级。

## 编译、测试与安全输入

- GCC 与 Clang 使用 JSON 中统一的严格 C99 选项并把告警视为错误；不得单独降低本地门禁。
- 每个公开头文件分别以 C99 和 C++17 编译；本地与持续集成使用同一脚本和配置。
- 测试围绕公开行为，覆盖正常流程、最小/最大容量、非法参数、错误状态、资源耗尽和故障注入。
- 不可信输入先验证长度、版本、校验和、范围和输出容量，再写业务对象。
- 字符串必须在可访问范围内找到并保留结尾空字符；不得把不明确边界的字符串写入固定数组。
- 二进制长度使用显式 `size_t`，不得用 `strlen` 计算；字符输入函数返回值必须用 `int` 接收。
- 格式化输出使用编译期固定格式串、与参数类型匹配的格式宏，并检查负返回值和截断。
- 禁止把用户输入当格式串，禁止 `system`、`popen` 等命令执行路径。
- 禁止定义、重定义或取消定义标准保留标识符；除非有批准的特殊需求，不使用嵌入式汇编。

## 完成定义

变更只有在以下证据同时成立时才完成：规则目录仍为 139/139 且未分类为 0；黄金 demo 可重复生成；
正例 lint 通过且负例命中预期规则；GCC 和 Clang 严格 C99 零告警；四个头文件分别通过 C99/C++17；
行为、边界和故障注入测试通过；`c-kit`、snippets 和规则目录 JSON 均通过各自 schema。
