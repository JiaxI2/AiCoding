# 实现计划：华为 C 规范全覆盖黄金示例

Plan Status: Approved

## 阶段

1. 使用 Docker 中的官方工具将 PDF 转换为 Markdown，校验章节、规则标题、代码块和可读性。
2. 建立完整规则目录，每条 PDF 规则都有唯一 ID、来源和执行位置。
3. 实现 Option D 的简单入口，并将 3 个高级模块整理到公开 `advanced/`。
4. 扩展确定性生成器，一次生成简单入口、导航和全部高级黄金文件。
5. 保持 `c-kit.json` 表达规则门禁；新增 VS Code 兼容 `c-snippets.json`、schema 和渲染入口。
6. 扩展 lint，并为可机器检查规则建立正向与负向 fixture。
7. 加入 GCC、Clang、C++17 头文件和运行时行为门禁。
8. 运行 Go 测试、生成一致性、规则覆盖和文档链接验证。
9. 以复杂函数编号总览、逻辑段意图和分支副作用重构注释层级，并明确语义规则的人工评审边界。

## 成功标准

- Markdown 参考包含全部 15 章以及所有“原则/规则/建议”标题，人工抽查可读；
- 规则目录没有未分类 PDF 条款；
- 生成根目录只含简单 `demo.c/.h` 与 `advanced/`；高级目录含 3 对 `.c/.h`、导航和测试；
- 全部静态前置声明仅含原型，定义处有完整函数注释；
- 复杂函数使用编号控制流总览，主要逻辑段之间有空行和段前意图注释；
- 非显然 `if/else` 与有实际处理的 `case/default` 说明领域意图，连续空 case 不强制重复注释；
- 未知工号不输出占位行，代码内修改历史默认禁用；
- `c-kit`、snippets 和规则目录 JSON 通过各自 schema；
- lint 正向样例零诊断、负向 fixture 命中预期规则；
- GCC/Clang 严格 C99 零告警，公共头文件通过 C99/C++17 独立编译；
- 行为测试通过，无动态内存、VLA、递归、无界循环和阻塞调用。

## 回滚

`c-userstyle-kit` 不是 Git 仓库，因此每次跨文件变更前保存明确文件清单；回滚以恢复对应文件内容或移除本次新增产物为边界，不运行破坏性清理命令。
