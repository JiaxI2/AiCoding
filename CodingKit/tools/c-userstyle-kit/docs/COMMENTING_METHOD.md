# C99 注释方法：从控制流总览到分支意图

本方法把华为 C 语言编程规范第 8 章的“自解释代码、准确注释、意图层注释和相邻排版”与实际
PDO 映射代码的阅读需求结合起来。目标不是提高注释数量，而是让维护者先看到函数骨架，再沿着
业务分支理解约束和副作用。

## 1. 先让代码自解释

注释不能修复职责混乱、命名含糊或嵌套过深。出现以下情况时先重构，再补注释：

- 一个函数混合对象查找、合法性检查、地址解析和结果发布等多个可独立命名的职责；
- 注释只能逐句翻译 `if`、赋值或循环；
- 阅读者必须在多个远距离分支之间反复跳转，才能确认一次业务操作是否完整。

函数数量不设目标值。只有能够形成独立职责、降低主流程认知负担，或消除真实重复的逻辑才提取
私有函数；一次使用且只有一两行的选择表达式通常保留在调用处。

## 2. 复杂函数先给控制流总览

当函数包含三个及以上主要逻辑段、多轮遍历、状态分派或“校验后统一发布”事务时，在定义处的
`@details` 使用编号概述控制流。编号对应真实职责，不要求使用“第一阶段”等固定词。

```c
/**
 * @brief 根据分配表生成过程数据映射。
 *
 * @details
 * 1. 校验分配数量，拒绝超出支持范围的配置。
 * 2. 扫描接收分配表，更新轴激活状态并累计输出位宽。
 * 3. 扫描发送分配表，累计输入位宽。
 * 4. 全部检查成功后统一发布输入和输出字节数。
 */
```

总览描述“函数为什么按这个顺序工作”，不复制变量名和每条语句。编号应能在函数体中找到对应的
逻辑块；代码顺序变化时必须同步更新总览。

## 3. 函数体按逻辑段组织

相对独立的业务逻辑段采用以下排版：

```c
/* 先验证外部长度，确保后续固定偏移读取均在帧内。 */
if (frame_length != EXPECTED_FRAME_SIZE)
{
    return RESULT_INVALID_LENGTH;
}

/* 所有字段验证通过后再统一发布，失败路径不留下部分结果。 */
message->command = command;
message->value = value;
```

要求：

- 段前注释与上一代码块之间留一个空行；
- 注释与目标代码相邻、缩进一致；
- 注释说明目的、约束、业务顺序或副作用，不写“长度不等于就返回”之类机械复述；
- 连续赋值属于同一个逻辑事务时使用一条段注释，不逐字段堆叠注释。

## 4. `if/else` 注释到领域意图

非显然判断需要说明条件在业务中的意义，以及分支将保留或改变什么：

```c
/* 只有完整批次才能发布平均值；否则保留调用方原有输出。 */
if (context->dma_pending && (context->sample_count > 0U))
{
    *average = DEMO_CalculateAverage(context->samples, context->sample_count);
    context->dma_pending = false;
}
else
{
    /* 不完整批次不清除通知，允许后续周期在数据就绪后继续处理。 */
    result = DEMO_RESULT_NO_DATA;
}
```

以下情况不强制为每个分支分别写注释：

- 参数无效后立即返回的简单 guard；
- 条件和动作已经由领域名称完整表达；
- `else` 没有业务动作。此时应删除空 `else`，不要为了容纳注释保留空分支。

## 5. `switch/case` 注释到模式和副作用

每个包含实际处理的 `case/default` 在首条语句前写一条中文意图注释：

```c
switch (mode)
{
    case MODE_POSITION:
        /* 位置模式发布状态字和实际位置，并按该布局推进缓冲区游标。 */
        DEMO_WritePositionFrame(cursor, axis);
        break;

    case MODE_DISABLED:
    case MODE_RESERVED:
        /* 两种不可用模式共用拒绝策略，不改变当前过程数据。 */
        result = RESULT_INVALID_MODE;
        break;

    default:
        /* 未知模式表明配置已损坏，收敛为确定错误而不是猜测布局。 */
        result = RESULT_INVALID_MODE;
        break;
}
```

连续且没有处理语句的 `case` 标签不要求逐个注释。若一个分支处理后有意进入下一分支，必须在
两个标签之间写明贯穿原因；没有说明的隐式贯穿视为错误。

## 6. 简单宏使用单行块注释

对象式常量宏只需要一类简要说明时，使用单行普通块注释：

```c
/* 接收方向允许配置的最大 PDO 映射数量。 */
#define PDO_RX_MAPPING_MAX_COUNT 2U
```

只有需要导出参数、返回值或复杂用法时才使用 Doxygen。头文件保护宏不需要业务注释；不得为了
展示规则向黄金代码添加没有功能用途的宏。

## 7. 文件元数据和修改历史

文件头保留 `@file`、`@brief`、版权、日期、作者、文件内容、主要功能和文件关系。工号只有
在用户提供真实值时才出现；未知时直接省略，不写“不适用”“未提供”或类似占位文本。上游规范
中的文件版本字段由本 Kit 显式覆盖：源码不写资产 `@version`，版本统一由 `MANIFEST.json`、
资产文档、`CHANGELOG.md`、Tag/Release 权威面管理。

代码内修改历史默认禁用。开发变更由 Git 和 `CHANGELOG` 记录；维护发布或缺陷修复工程只有在用户
显式确认后才能启用源文件修改历史。该策略是项目对华为规范 8.2 的显式覆盖，原始 PDF/Markdown
仍保持不变。

## 8. 缩写、语言和项目一致性

- 专业缩写首次出现时给出中文含义、英文全称和缩写，例如直接内存访问
  （Direct Memory Access，DMA）。业界通用或项目内已登记的标准缩写可以直接使用。
- 同一项目统一 Doxygen 标签、普通块注释、术语、标点和分支说明方式，不在相邻模块中混用相互
  冲突的格式。项目既有约定高于个人偏好。
- 默认使用准确、简洁的中文说明领域意图。国际协作项目应由项目级 `AGENTS.md` 或配置明确选定
  中文、英文或双语策略；一旦选定，同一层级保持一致，不逐文件随意切换语言。
- 禁止在表达式或条件中间插入注释。需要解释复合条件时，把意图注释放在完整表达式上方，或先
  提取为具有领域名称的布尔变量，避免注释打断求值关系。

## 9. 证据边界

lint 可以稳定检查注释形式、位置、空行、连续空 `case` 例外、贯穿说明和配置化文件元数据。
普通逻辑段是否确有独立职责、分支是否属于“非显然”、注释是否准确表达领域意图，以及函数是否
应先重构，都属于 manual 评审。lint 只能检查这些注释已经存在时的稳定结构，或由明确阈值触发的
控制流总览，不能用关键词正则伪装成语义验证。
